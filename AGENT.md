# AGENT.md

Guidance for future automated coding agents working on SnapEx.

## Project summary

SnapEx is a local Visual Studio Code extension for backing up and restoring installed VS Code extensions. It creates one final zip package per backed-up extension. Each final package contains a timestamped backup folder, a `backup-index.json`, and a nested per-extension archive with extension files, contributed VS Code settings, storage folders, selected external state files, and metadata.

The extension is published locally under:

- Extension id: `local-tools.snapex`
- Display name: `SnapEx`
- Activity Bar container id: `snapex`
- Main view id: `snapex.actions`
- Main source file: `src/extension.ts`

Older builds used `local-tools.extension-state-backup`; scripts and uninstall flows still account for that historical id.

## Important user intent

The user wants SnapEx to behave like a portable extension-state backup tool. Recent user-requested behavior includes:

1. The extension must be called SnapEx.
2. `snapex-icon.png` must be used for the VS Code Activity Bar icon.
3. Backups must include extension state where reasonably possible.
4. Continue's external config file at `~/.continue/config.yaml` must be backed up and restored.
5. The final backup product must be a single outer zip package, not a leftover uncompressed folder.
6. The uncompressed staging folder must be deleted after the final zip is written.
7. Backup package names must use the local machine time in this pattern:

   ```text
   vscode-extension-backup-<extension-name-and-version>_YYYYMMDDHHMMAM/PM.zip
   ```

   Example:

   ```text
   vscode-extension-backup-Continue.continue-2.0.0_202607080405PM.zip
   ```

8. Restore should accept either the final outer `vscode-extension-backup-...zip` package or the nested per-extension zip directly.
9. `update-extension.sh` should automate update, rebuild, package, reinstall, and verification. It should leave only the VS Code window reload as a manual step.

## Repository layout

Key files:

- `package.json`: extension manifest, commands, settings, Activity Bar contribution, scripts, current version.
- `src/extension.ts`: backup, restore, naming, archive, configuration, storage, external-state, and safety logic.
- `test/regression-test.js`: comprehensive mocked-extension-host regression suite.
- `test/smoke-test.js`: older narrower smoke test kept for comparison.
- `update-extension.sh`: local update/build/package/install/verify helper.
- `README.md`: user-facing usage, update, restore, and test documentation.
- `CHANGELOG.md`: version notes.
- `snapex-icon.png`: extension and Activity Bar icon.
- `.vscodeignore`: VSIX packaging exclusions.

## Commands to run before committing

Use these from the repo root:

```bash
npm install
npm run compile
npm test
npm run package
```

`npm test` is the expected full regression command. It compiles TypeScript and runs `test/regression-test.js` with mocked VS Code APIs and temporary filesystem fixtures. It does not require a real VS Code window.

Useful narrower commands:

```bash
npm run test:regression
npm run test:smoke
```

The VSIX generated by `npm run package` should match the version in `package.json`, for example `snapex-0.1.4.vsix`.

## Local install/update workflow

The preferred local user workflow is:

```bash
bash update-extension.sh
```

For a local branch that has diverged and should be force-aligned with GitHub:

```bash
bash update-extension.sh --reset-to-origin
```

The script should:

1. Fetch `origin/main` and tags.
2. Fast-forward, or reset when explicitly requested.
3. Inspect the current `package.json` version.
4. Compare it with the latest semver release/tag it can see.
5. Run dependency install and tests.
6. Package the current VSIX.
7. Uninstall old SnapEx extension ids.
8. Install the packaged VSIX with `--force`.
9. Verify installation by running:

   ```bash
   code --list-extensions --show-versions | grep '^local-tools.snapex@'
   ```

After the script completes, the user still needs to reload VS Code.

## Versioning expectations

When behavior changes in a way users need to reinstall, bump `package.json` version and update `CHANGELOG.md` and `README.md` as needed.

This matters because VS Code can appear to ignore a local VSIX reinstall when the extension id and version are unchanged. Version bumps make local testing and reinstall behavior clearer.

## Backup architecture notes

Backup entry points in `src/extension.ts`:

- `backupAllExtensions(context)` backs up all eligible installed extensions.
- `backupSelectedExtension(context)` backs up one selected extension.
- `createExtensionBackup(context, extension, runDirectory, settings)` creates the nested per-extension zip.
- `finalizeBackupRunDirectory(runDirectory)` creates the final outer zip and deletes the staging folder.
- `backupRunDirectoryNameFor(extension, timestamp)` controls final package naming.
- `formatLocalBackupTimestamp(date)` intentionally uses local machine time, not UTC.

Each final outer package should contain a single top-level folder named like:

```text
vscode-extension-backup-<extension-name-and-version>_YYYYMMDDHHMMAM/PM/
```

Inside that folder:

```text
backup-index.json
<extension-id>-<version>.zip
```

The nested per-extension zip may contain:

```text
manifest.json
configuration/configuration.json
extension/
globalStorage/
workspaceStorage/current/
externalState/home/
metadata/external-state.json
metadata/extension-file-modes.json
```

## Restore architecture notes

Restore entry points in `src/extension.ts`:

- `restoreFromZip(context)` is the command handler.
- `openRestorableBackupZip(zipPath)` accepts either the final outer package or the nested extension archive.
- `readManifest(zip)` validates schema support and extension identity.
- `restoreExtensionFiles(context, zip, manifest)` overwrites installed extension files.
- `restoreConfiguration(zip)` replays contributed VS Code configuration values.
- `restoreDirectoryFromArchive(zip, prefix, target)` restores global/workspace storage folders.
- `restoreExternalState(zip)` restores known external state files under the current user's home directory.

Restore currently supports backup schema versions from `MIN_SUPPORTED_BACKUP_SCHEMA_VERSION` through `BACKUP_SCHEMA_VERSION`.

Keep restore conservative. It should prompt before overwriting when `extensionStateBackup.confirmBeforeRestore` is enabled.

## External state behavior

SnapEx only captures known, extension-owned external config files. At the moment, the explicit external-state rule is:

```text
continue.continue -> ~/.continue/config.yaml
```

This is implemented in `getKnownHomeRelativeExternalStatePaths(extension)`.

When adding more external-state paths:

1. Only include paths that are clearly owned by that extension.
2. Prefer files under the user's home directory.
3. Avoid broad directories unless there is a strong reason and tests cover it.
4. Update README limitations if the new behavior changes what users can expect.
5. Add regression coverage for both backup and restore.

Tests may set `SNAPEX_TEST_HOME` so external-state behavior does not touch the real home directory.

## Safety and path traversal

Archive extraction must never allow zip entries to write outside the intended target directory.

Important helpers:

- `safeJoin(rootDirectory, relativePath)` validates extraction targets.
- `normalizeHomeRelativePathForArchive(relativePath)` rejects absolute paths and `..` segments for external-state paths.
- `fromArchivePath(filePath)` and `toArchivePath(filePath)` normalize path separators.

When changing archive extraction or external-state restore, add or update regression cases for malicious `../` entries.

## Known limitations to preserve in docs

VS Code does not expose everything about other extensions through public APIs. SnapEx should not claim to back up:

- SecretStorage values.
- Passwords, auth tokens, OS keychain entries, or authentication sessions.
- Every private Memento/globalState record when an extension stores it in VS Code's shared internal state database.
- Workspace storage for workspaces that are not open during backup.
- Arbitrary custom files referenced from extension settings outside the known external-state allowlist.

Use “best effort” language for state capture unless a path is explicitly implemented and tested.

## Regression test expectations

The regression suite should continue to verify at least these behaviors:

1. Commands are registered.
2. Built-in extensions are excluded by default.
3. Final outer zip naming uses local `YYYYMMDDHHMMAM/PM` timestamps.
4. The uncompressed staging folder is deleted after final zip creation.
5. `backup-index.json` and nested per-extension archives are created.
6. Extension files, configuration, global storage, current-workspace storage, file modes, and external state are captured.
7. Restore works from both the outer package and nested archive.
8. Global, workspace, and workspace-folder settings are replayed.
9. Restore cancellation leaves files/settings/state untouched.
10. Malicious archive entries are rejected.
11. Older-compatible external-state archives without `metadata/external-state.json` still restore by deriving metadata from `externalState/home/`.
12. Open-backup-folder behavior works for configured and fallback paths.

## Git and sync notes

The user's local checkout has previously diverged from GitHub after remote commits were added. When that happens and their working tree is clean, the recommended fix is:

```bash
cd ~/Projects/snapex
git fetch origin --tags
git switch main
git reset --hard origin/main
```

The project script also supports:

```bash
bash update-extension.sh --reset-to-origin
```

Do not suggest a plain `git pull` for a divergent branch unless a merge or rebase strategy has been intentionally chosen.

## Coding style notes

- The extension is TypeScript targeting VS Code `^1.92.0`.
- Keep most implementation in `src/extension.ts` unless the file becomes difficult to maintain.
- Use `fs/promises` and path-safe helpers consistently.
- Use `adm-zip` for archive creation and reading.
- Keep command ids stable unless there is a migration reason.
- Preserve backward compatibility with `local-tools.extension-state-backup` in uninstall/update flows where relevant.
- Prefer explicit, tested behavior over broad filesystem scans.
- Update tests and README alongside user-visible behavior changes.

## Commit checklist

Before committing:

1. Confirm `package.json` version if the change affects installed behavior.
2. Update `CHANGELOG.md` for user-visible behavior changes.
3. Update `README.md` for changed commands, archive format, install, restore, or test behavior.
4. Run `npm test` where possible.
5. Run `npm run package` where install/package behavior changed.
6. Make sure no generated `.vsix`, temp backup zips, or decompressed backup folders are accidentally committed unless intentionally used as fixtures.
